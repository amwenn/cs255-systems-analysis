DriverPass Project Reflection
Project Summary

The DriverPass project involved designing a system for a client named DriverPass, a company focused on helping students better prepare for their driving tests. The client wanted a web-based system that would allow students to register for training packages, schedule driving lessons, access online learning materials, and take practice tests. The system also needed to support office staff, instructors, and administrators while maintaining security and tracking user activity. My role was to gather the client’s requirements, document them clearly, and design system models that accurately reflected their needs.

What I Did Well

One area I believe I did particularly well was translating the client’s business needs into structured documentation and visual system models. I carefully analyzed the interview transcript and ensured that key requirements such as online scheduling, role-based access, test progress tracking, and reservation auditing were represented in my use case, activity, sequence, and class diagrams. I also focused on clarity and organization so that the system design could be easily understood by both technical and nontechnical audiences.

What I Would Revise

If I could revise one part of my work, I would further refine the class diagram to make it even more streamlined and focused on core system entities. While the diagram accurately reflects the system structure, I would improve it by simplifying some relationships and ensuring perfect consistency between attributes, multiplicity, and terminology across all diagrams. This would enhance readability and alignment between documentation sections.

Interpreting User Needs

To interpret the user’s needs, I carefully reviewed the client interview transcript and identified both explicit and implied requirements. For example, the client emphasized tracking who modified reservations, which led me to incorporate activity logging into the system design. I also ensured that scheduling logic reflected real-world constraints such as driver and vehicle availability. Considering user needs is critical because a system that does not reflect actual workflows or expectations will fail, even if it is technically sound. Designing with the user in mind ensures usability, efficiency, and long-term success.

My Approach to Designing Software

When designing software, I begin by clearly defining the problem and identifying stakeholders. I break the system into functional requirements and model interactions visually before thinking about implementation details. Using UML diagrams helps me validate logic and relationships early in the design process. In the future, I plan to continue using structured modeling techniques, iterative refinement, and regular validation against client requirements to ensure that systems remain aligned with business goals.
